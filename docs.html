Project Overview

Precision Workz is a single-page agency website served by a lightweight Node.js HTTP server. There is no framework (no Express, no React). The entire frontend lives in one file: index.html. The backend lives in one file: server.js. All data is stored as flat JSON files on disk.

Frontend

  • Single index.html (~3,700 lines)
  • Vanilla CSS — no Tailwind, no Bootstrap
  • Vanilla JS — no jQuery, no bundler
  • Phosphor Icons (CDN)
  • Orbitron + Inter (Google Fonts)
  • Stripe.js v3 (CDN)
  • Google Identity Services (CDN)

Backend

  • Node.js built-in http module
  • No Express — routes handled manually
  • Stripe npm package for checkout
  • Data stored as flat JSON files
  • Runs on port 4000 (configurable via PORT env)
  • .env parsed manually at startup

File Structure

precision-workz/
├── server.js          ← HTTP server + all API routes
├── index.html         ← Entire frontend (HTML + CSS + JS)
├── .env               ← Secrets (Google Client ID, API key)
├── package.json       ← npm config (only dependency: stripe)
├── admins.json        ← List of admin email addresses (auto-created)

Note: admins.json, clients.json, and requests.json are created automatically the first time data is written. They do not need to be manually created. If they don't exist, the server falls back to safe defaults.

Starting the Server

node server.js
# or
npm start

The server reads .env at startup. If a key is already set in the environment, the .env value is skipped (no overwrite). It then tries to require('stripe'). If Stripe is not installed, it logs a warning and continues — the site still works, just without checkout.

Port conflict: If you see EADDRINUSE: address already in use :::4000, a previous Node process is still running. Kill it: Stop-Process -Name "node" -Force (PowerShell) then restart.

Login System

Authentication is client-side only. The server never issues session tokens. The browser stores user data in localStorage as a JSON object under the key pw_user. There are two ways to sign in:

Method 1 — Google OAuth (Google Identity Services)

1
Page load

The frontend fetches /api/config to get the Google Client ID, then calls google.accounts.id.initialize() with handleGoogleCredential as the callback.

2
Sign-in modal opens

openSignIn() shows the modal, then after an 80ms delay (to ensure the container is visible), calls google.accounts.id.renderButton() to inject the real Google button into #google-signin-btn-wrap.

3
User picks a Google account

Google calls handleGoogleCredential(response). The function base64-decodes the JWT payload (no signature verification — this is client-side only) and extracts name, email, picture.

4
User data saved

The decoded user object is stored via setUserData(user) → localStorage.setItem('pw_user', JSON.stringify(user)). updateAuthUI(user) is called to update the nav.

One-shot render: The Google button is only rendered once per page load (_gBtnRendered flag). If the modal is closed and reopened before GSI loads, the button may not appear. A page refresh always fixes it.

Method 2 — Email Fallback (no password)

If a user doesn't want to use Google, they can type their name + email in the fallback form. emailSignIn() validates the format and stores the same pw_user object in localStorage (with an empty picture field). No server call is made.

Security note: The email fallback has no password, no OTP, and no server verification. Anyone who knows your email can sign in as you in the browser. This is intentional for the demo phase — it's gated by the admin panel assigning a packageType before any real access is granted.

Session Persistence

On every page load, the init block runs:

var user = getUserData(); // reads localStorage
if (user) {
  updateAuthUI(user);
  // background-syncs subscription/packageType from server
  fetch('/api/client-status?email=' + user.email)...
}

This means a user stays "logged in" indefinitely (until they sign out or clear localStorage). The background sync ensures their subscription tier is always current from the server.

Sign Out

signOut() calls clearUserData() → localStorage.removeItem('pw_user'), then updateAuthUI(null) to reset the nav.

User Object Shape

{
  name:        "John Smith",
  email:       "john@example.com",
  picture:     "https://lh3.googleusercontent.com/...",  // empty string for email login
  sub:         "gold",        // null | "basic" | "gold"
  billing:     "monthly",     // "monthly" | "annual"
  packageType: "titanium"     // null | "starter" | "premium" | "diamond" | "titanium"
}

Admin System

The admin panel is a full-screen overlay accessible only to verified admin emails. All admin API routes require an adminEmail parameter that is checked server-side against admins.json.

Admin Hierarchy

How the Panel Opens

1
Login detected

After sign-in, checkAdminBtn(user) calls GET /api/admin/data?adminEmail=. If the response is not an error (200 OK), the Admin button is shown in the nav.

2
Panel opened

openAdmin() adds .open to #admin-overlay and calls adminLoadData().

3
Data loaded

adminLoadData() fetches /api/admin/data (clients + admins) and adminLoadRequests() fetches /api/admin/requests in sequence.

Admin Panel Sections

Stats Bar — Total clients · Basic count · Gold count · Active packages (has packageType) · Open support requests
Add / Update Client — Form to set email, name, subscription tier, billing cycle, package, and internal notes. Calls POST /api/admin/set-client. Also used to edit existing clients (click "Edit" in the table to pre-fill the form).
Developer Access — Lists all admin emails. Primary admin sees a Remove button next to each developer. Primary admin also sees the Add Developer input. Calls POST /api/admin/add-admin / POST /api/admin/remove-admin.
All Clients Table — Searchable table of all entries in clients.json. Columns: email, name, subscription pill, billing, package pill, notes, date added, Edit/Remove actions.
Support Requests — All entries from requests.json, filterable by status (All / Open / Resolved). Each card shows: status, type, date, customer name + email, request details, Reply button (opens mailto), and Mark Resolved / Reopen button.

Security model: Admin verification is email-only with no cryptographic token. The server trusts whatever email the client sends. For a private internal tool this is fine, but do not expose admin routes to the public internet without adding token-based auth.

Checkout Flow (Gated)

1
Subscribe button clicked

openPayModal(plan, name, price) is called. First checks: is the user signed in? If not → opens sign-in modal, stores pending plan in window._pendingPayPlan.

2
Customer check

If user is signed in but user.packageType is null (not a paying customer yet) → shows a toast: "You need an active package before subscribing." The subscribe flow stops here.

3
Step 1 — Confirm info

Pre-fills the user's name and email. User confirms or edits, then clicks Next. payStep1Next() validates fields, saves any edits back to localStorage, and advances to Step 2.

4
Step 2 — Stripe card

initStripe() mounts a Stripe Card Element into #stripe-card-element. The Stripe dark theme is applied via the Appearance API.

5
Submit

submitPayment() calls POST /api/create-checkout with the plan key and user email. The server creates a Stripe Checkout Session and returns {url}. The browser redirects to Stripe's hosted checkout page.

6
Return

On success: redirected to /?checkout=success → site shows a green toast "Subscription active — welcome aboard!". On cancel: /?checkout=cancel → purple toast "Checkout cancelled."

Completing Stripe Setup

  1. Go to dashboard.stripe.com/test/products
  2. Create 4 subscription products matching the plans above
  3. Copy each price_xxx ID from the dashboard
  4. Paste into server.js in the PRICES object:
const PRICES = {
  'basic-monthly': 'price_xxx',
  'basic-annual':  'price_xxx',
  'gold-monthly':  'price_xxx',
  'gold-annual':   'price_xxx',
};

Pending pay plan: If a user clicks Subscribe, gets routed to sign-in, and then signs in — the pending plan is stored in window._pendingPayPlan. After sign-in completes, _resumePendingPay() automatically reopens the payment modal with the original plan.

Email System

There is no email server. All email functionality uses mailto: links that open the user's default email client. This means:

  • No emails are sent automatically from the server
  • The user has to click Send in their own email client
  • No email confirmation, no auto-reply, no delivery guarantee

Contact Form (submitContact())

Collects: first name, last name, email, package selection (custom dropdown), message, optional promo code. Builds a mailto:precizionworkz@gmail.com link with subject and body pre-filled, then navigates to it. The promo code (if entered and validated via "Apply") is included in the email body.

Support Request Reply (Admin Panel)

Each support request card in the admin panel has a Reply button. It opens a mailto: link pre-addressed to the customer's email with a subject line of [Precision Workz] Re: {request type} and a starter greeting in the body.

Future: Real Email

To send emails automatically (e.g., on new request submission), you would add nodemailer or use a service like SendGrid/Resend. The server already has the request data and owner email constant OWNER_EMAIL = 'precizionworkz@gmail.com' ready to wire up.

API Routes

All routes are handled in server.js via the handleAPI() function. No Express — paths are matched with simple if statements.

Method + PathAuthWhat it does
GET/api/config Public Returns {stripeKey, googleClientId} — used by the frontend on load
GET/api/pub-key Public Returns {key: STRIPE_PUB} — legacy alias
GET/api/client-status Public Query param: ?email=. Returns the client record from clients.json or null
POST/api/request Public Submit a support request. Body: {email, name, type, details, sub}. Saves to requests.json
POST/api/create-checkout Public Body: {plan, email}. Creates Stripe Checkout Session, returns {url}
POST/api/demo Public Logs a demo booking request to console. Body is any JSON
GET/api/admin/data Admin email Query: ?adminEmail=. Returns {clients, admins, primaryAdmin}
GET/api/admin/requests Admin email Query: ?adminEmail=. Returns {requests, ownerEmail}
POST/api/admin/set-client Admin email Add or update a client record. Body includes all client fields
POST/api/admin/remove-client Admin email Delete a client. Body: {adminEmail, targetEmail}
POST/api/admin/update-request Admin email Change a request's status. Body: {adminEmail, id, status}
POST/api/admin/add-admin Primary admin only Grant admin access to a new email. Body: {adminEmail, newEmail}
POST/api/admin/remove-admin Primary admin only Revoke developer access. Body: {adminEmail, targetEmail}

All non-API paths fall through to static file serving. If the file doesn't exist, the server returns index.html (SPA-style fallback).

Mobile Nav — Slide-in overlay. openMobileNav() / closeMobileNav().

An object keyed by lowercase email address. See the shape in the Admin System section above.

Login Logs

Login logs are not yet implemented. The server does not record when a user signs in, what IP they connected from, or what email was used. All auth happens client-side (localStorage only).

What Exists Today

  • Server logs to console: [Request submitted] {type} from {email} when a support request is saved
  • Server logs to console: [Demo Request] {data} when a demo is booked
  • No sign-in events are logged anywhere

How to Add Login Logging (Future Work)

Create an auth-log.json file and a POST /api/auth-log endpoint. Call it from handleGoogleCredential() and emailSignIn() in index.html after a successful sign-in:

// In handleGoogleCredential / emailSignIn, after setUserData():
fetch('/api/auth-log', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    email: user.email,
    method: 'google',  // or 'email'
    timestamp: new Date().toISOString(),
    ua: navigator.userAgent
  })
});

On the server, append to an array in auth-log.json. This gives you a timestamped record of who signed in and how. Adding req.socket.remoteAddress on the server side gives you the IP.

Known Bugs & Incomplete Features

🔴 Stripe Checkout Always Returns 402

The PRICES object in server.js has null for all four plan keys. Stripe products and prices have not been created in the dashboard yet. Clicking Subscribe hits POST /api/create-checkout, the server finds no price ID, and returns a 402 "setup_required" error. Fix: create 4 subscription products in the Stripe test dashboard and paste the price_xxx IDs into PRICES.

🔴 No Real Email Sending

The contact form and admin Reply button both use mailto: links. If the user's default email client is not set up, nothing happens. No automatic notification is sent when a support request comes in. Fix: add nodemailer or a transactional email API (SendGrid, Resend, etc.) and wire it into POST /api/request.

🟡 Google Button One-Shot Render

The Google sign-in button is rendered once per page load (_gBtnRendered flag). If the sign-in modal is opened before Google Identity Services has finished loading, the button won't appear. Closing the modal and reopening it won't retry (flag is already true). Fix: remove the flag or check if the button container is actually populated before setting the flag.

🟡 Subscription Status Only Syncs on Dashboard Open

The background sync on page load calls /api/client-status and updates the CTA banner. But if the admin changes a client's package while that client's tab is open, the client won't see the change until they close and reopen their dashboard or refresh the page.

🟡 No Login Logs

Sign-in events are not recorded anywhere on the server. There is no audit trail of who has accessed the admin panel. See the Login Logs section above for how to implement this.

🟡 Admin Auth Is Email-Only (No Token)

The server trusts whatever adminEmail value the client sends. Anyone who knows an admin email address can make admin API calls directly (e.g., via curl or Postman). The UI prevents this for normal users, but there is no cryptographic guarantee. Fix: issue a signed token (JWT with server secret) on admin login, verify it on each admin request.

🟡 Port Conflict on Restart

If the server is stopped with Ctrl+C but the Node process doesn't fully terminate (happens sometimes on Windows), restarting gives EADDRINUSE :::4000. Fix: Stop-Process -Name "node" -Force in PowerShell before restarting, or use nodemon for development.

🟢 "Start Another Project" Only Shows After Dashboard Open

On page load, the background client-status fetch updates the user's packageType, which should trigger the CTA banner + nav button update. However, if the server is slow, the nav CTA may briefly show "Start a Project" before switching. Not a functional bug — just a brief flash.

🟢 Spotify Embed Requires Internet

The Spotify iframe embed only works when connected to the internet and when Spotify's CDN is reachable. On localhost without internet, it will show a blank area. Not fixable — by design.
